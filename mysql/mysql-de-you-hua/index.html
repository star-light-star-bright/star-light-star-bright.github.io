SQL优化

1、插入数据

insert优化

批量插入
insert into table_name values(value),(value)
大批量插入数据

一次性插入大批量数据，使用insert性能低，使用Mysql提供的load指令进行插入。

#1.连接服务端时，加上参数 local-infile 
mysql --local-linfile -u root -p 

#设置全局参数local_infile为1，开启从本地加载文件导入数据的开关
set global local_infile=1;

#执行load指令将准备好的数据,加载到表结构中
load data local infile 'root/sql1.log into table 'table_name' fields terminated by ',' lines terminated by '\n';
手动提交事务
start transaction;
insert into table_name values(value),(value);
insert into table_name values(value),(value);
commit;
主键顺序插入

顺序插入数据性能高于乱序插入

8 1 5 6 46  2 6  584 
1 2 3 4 5 6 7 8 9

2、主键优化

数据组织方式

表数据根据主键顺序组织存放的，这种存储方式表成为索引组织表

页分裂

页可以为空，也可以填充一半，也可以填充100%，每个页包含了2-N行数据（如果一行数据多大，会行溢出），根据主键排序。

主键乱序情况插入

保证主键的顺序性，不会插入到新开的page，会插入到47后面

要插入到47后面，但是没有空间可以插入

会新建立一个page，并把原来需要插入的page进行拆分，变成两个，把后半部分移动到新开的page，并插入到他后面，但是这样的顺序破坏了有序性。

为了保证有序性，把page的指针从新设置

页合并

主键删除情况

当删除一行数据时，并没有真正的物理删除，只是记录被标记为删除并且它的空间变得允许被其他记录声明使用

当页中删除的数据记录达到MERGE_THRESHOLD（合并页的阈值，可以自己设置，默认page50%）innoDB自动寻找靠近的页看看是否可以将两个页合并并优化空间使用

主键设计原则

  1. 满足业务需求的情况下，尽量降低主键长度。

主键长度很长，二级索引表存储着主键值，查询时会占用大量io。

  1. 插入数据，尽量选择顺序插入，使用AUTO_INCREMENT自增主键，避免页分裂
  2. 尽量不使用用UUID做主键或者其他自然主键，如身份证。不仅长还没有顺序。
  3. 避免对主键的修改。

3、oreder by 优化

  1. 建立合适的索引，多字段排序时，遵循最左前缀法则。

  2. 尽量使用覆盖索引。

  3. 多字段排序，创建联合索引时的规则。

  4. 出现filesort大数据量排序时，适当增大排序缓冲区的大小，sort_buffer_size默认大小256K，占满了就到磁盘中排序。

优化时尽量优化到Using index，提高效率。

create index index_name on table_name(a,b);
#不指定默认都是正序索引

explain select id,age,phone from tb_user order by age,phone;
#全部走索引，正序，不触发filesort

explain select id,age,phone from tb_user order by age desc,phone desc;
#全部倒序走索引，反向扫描，不会触发filesort

explain select id,age,phone from tb_user order by age asc,phone desc;
#一个正一个倒，会出现filesort。可以单独创建一个索引：
create index index_name on table_name(a asc,b desc);

Using filesort

Using filesort:通过表的索引或者全表扫描，读取满足条件的数据行，在排序缓冲区sort buffer中完成排序操作，suo’you不是通过索引直接返回排序结果的排序都叫FileSort排序。

Using index

Using index：通过有序的索引顺序扫描直接返回有序数据这种情况即为using index ，不需要额外排序，操作效率高。

4、group by 优化

  1. 分组操作时，通过索引来提高效率。

  2. 分组操作，索引的使用也是满足最左前缀法则的。

5、limit 优化

查询limit 20000000,10 先排序20000010条数据，在获取最后10条数据，其他数据全部丢弃，查询代价非常大。

  1. 分页查询时，通过覆盖索引+子查询的方式进行优化。
explain select * from table_name a,(select id from table order by id limit 20000,10) b where a.id=b.id;
#子查询走主键索引
#外查询走聚集索引

6、count 优化

MyISAM引擎把一个表的总行数存到了磁盘上，因此执行count（*）的时候会直接返回这个数，效率很高。

InnoDB引擎，执行count(*)时，也需要全量读取数据在累积计数。

  1. 通过redis 自己记录行数。

count(主键)：遍历整张表，取出所有主键id，返回给服务层。 拿到主键后进行累加。

count(字段值)：看是否有not null 约束，没有就把值都取出来，进行判断是否为Null，不为Null，计数累加。有约束，取值后直接累加不判断。

count(1)

会遍历整张表，不会取值。返回的每一行放一个1进去进行累加。

count(*)

InnoDB做了优化，不会取值，直接累加。

效率排名：count(1)≈count(*)>count(主键id)>count(字段)

7、update 优化

开启事务更新数据时，根据建立的索引字段作为条件进行更新。

主键id作为更新条件时，是row clock （行锁）。行锁是对索引加的锁。

其他没有索引的字段作为更新条件时，是table clock （表锁）。一旦锁表了并发性能就降低
