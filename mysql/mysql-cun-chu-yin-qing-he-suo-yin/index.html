InnoDB

高可靠高性能的通用存储引擎，5.5版本以后是Mysql默认的存储引擎。

特点

高可靠：DML遵循ACID，支持事务，行级锁，提高并发访问性能。

高性能：支持外键 FOREIGN KEY 约束，保证数据完整性和正确性。

文件

表名.ibd:innoDB引擎每张表都会对应着这样的一个表空间，存储该表的表结构（frm，sdi）、数据和索引。

sdi：表结构数据

参数：innodb_file_per_table，默认一个表一个文件，这一通过这个参数设置。ibd包含sdi。

逻辑存储结构

TableSpece：表空间

表空间对应一个.ibd文件，一个Mysql实例可以对应多个表空间，用于存储记录、索引等数据。

Segment：段

段分为：数据段（Leaf node segment），索引段(Non-leaf node segment)，回滚段(Rollback segment)。InnoDB是索引组织表，数据段就是B+树的叶子节点，索引段即为B+树的非叶子节点。段用来管理多个Extent（区）

Extent：区

区，表空间的单元结构，每个区的大小为1M。默认情况下，InnoDB存储引擎页大小为16K，及一个区中一共有64个连续的页。

Page：页

InnoDB存储引擎磁盘管理的最小单元，每个页大小默认为16k。为了保证页的连续性，InnoDB存储引擎每次从磁盘中申请4-5个区

Row：行

InnoDB存储引擎数据按行进行存放。

隐藏列

Trx_id：每次对某条记录进行改动时，都会把对应的事务id赋值给trx_id隐藏列

Roll_pointer：每次对某条引记录进行改动时，都会把旧的版本写入到undo日志中，然后这个隐藏列就相当于一个指针，可以通过它来找到该纪录修改前的信息。

介绍

MyISAM是MySQL早期的默认存储引擎。

特点

不支持事务，不支持外键，支持表锁，不支持行锁，访问速度快。

文件

.MYD：数据

.MYI：索引

.sdi：表结构：json格式的文本数据

Memory

介绍

特点

内存存放，hash索引

文件

.sdi：表结构信息

引擎的特点和区别

innoDB支持事务，MyISAM不支持

innoDB支持行锁，MyISAM支持行锁

innoDB支持外键，MyISAM不支持外键

存储引擎的使用

根据应用场景选择不同的存储引擎。

innoDB：事务完整性和并发条件下数据一致性。

MyISAM：读数据和插入数据为主，很少更新和删除，对事务的完整性、并发性要求不高。这种引擎被nosql数据库替代了，MongoDB数据库。

MEMORY：将所有数据保存在内存中，访问速度快。表大小有限制，太大的表无法存放内存并保证安全性。被Redis取代了。

索引

1、索引介绍

索引（index）帮助Mysql高效的获取数据的**数据结构(有序)**。在数据之外，数据库还维护着满足特定查找算法的数据结构，这些数据结构以某种方式引用（指向）数据，在这些数据结构上 实现高级查找算法，这种数据结构就是索引。

优缺点

优点：

  1. 提高数据检索效率，降低数据库的IO成本
  2. 通过索引列对数据进行排序，降低数据排序的成本，降低CPU的消耗

缺点：

  1. 索引列暂用额外磁盘空间。
  2. 索引 加快了查询的效率，也同时降低了更新表的效率，DML语句执行效率底下。

2、索引结构

Mysql存储引擎和索引在存储引擎层实现，不同存储引擎不同的索引结构：

  1. B+

    常见的索引类型，大部分存储引擎都支持B+树索引

  2. Hash索引

    底层数据结构使用Hash表实现的，不支持范围查询，只能精确匹配索引列查询才有效

  3. R-Tree（空间索引）

    是MyISAM引擎的一个特殊索引类型，主要用于地理空间数据类型，通常使用较少

  4. Full-text

    一种通过建立倒排索引，快速匹配文档的方式。类似lucene,Solr,ES

B+树索引

对经典的B+Tree进行优化。原来B+Tree的基础上，增加了指向相邻叶子结点的链表指针，就形成了带有顺序指针的B+Tree，提高了区间访问的性能

Hash索引

采用hash算法，将键值转换成新的hash值，映射到对应的槽位上，存储在hash表中。

优点
  1. 查询效率高，通常只需要一次检索就可以了，效率通常要高于B+Tree索引。如果hash碰撞了还需要检索链表
缺点

  1. 只能等值匹配，不支持范围查询。

  2. 无法利用索引完成排序操作

  3. 将两个或更多的键值映射到相同的hash槽位上，就产生了hash碰撞，可以通过链表来解决。

3、索引分类

索引类型分类：

1、主键索引
2、唯一索引
3、常规索引
4、全文索引

索引存储形式分类：

innoDB存储引擎中，根据索引的存储形式，又分为以下两种：

1、聚集索引：

叶子节点下面挂着的是每一行的数据。

规则定义：

如果有主键，主键索引就是聚集索引

如果没有主键，将唯一的UNIQUE索引作为聚集索引。

如果没有主键也没有唯一索引，InnoDB自动生成rowid作为隐藏的聚集索引。

2、二级索引：

叶子节点下面数据是每行数据对应的id。

规则定义：

聚集索引之外的就是二级索引。

回表查询

指的是查询数据的一种流程，查询时先找二级索引找到对应的主键值，再根据主键值到聚集索引中拿到具体的行数据。

4、索引语法

创建索引

一个索引关联单个字段，被称为单列索引。

一个索引关联两个或者多个字段，称为联合索引或者组合索引。

create [UNIQUE|FULLTEXT] INDEX index_name ON table_name(index_col_name,....)
#一个索引可以关联多个字段
#常规索引不需要指定关键字

查看索引

show INDEX From table_name;

删除索引

DROP INDEX index_name ON table_name;

2、时间分析

慢查询日志

慢查询日志中记录了执行时间超过了指定参数的Sql语句的日志。

Mysql的慢查询日志默认没有开启，需要在MySQL的配置文件 /etc/my.cnf 中配置

show variables like 'slow_query_log'
#查看慢查询日志开启状态

vim /etc/my.cnf

slow_query_log = 1;
#开启慢查询日志功能
long_query_time = 2;
#设置慢查询日志的时间，这里设置为2秒

配置完成后重新启动MySQL服务器，查看 /var/lib/mysql/localhost-slow.log。只会记录超过时间的sql。

profile详情

查看每一条sql的耗时和各个阶段消耗的时间

select @@have_profiling;
#查看MySQL是否支持profile操作

profiling默认是关闭的，可以通过set在global/session级别开启profiling；

set profiling = 1;

查询profiling

show profiles;
#查看每一条SQL的耗时情况
show profile for query query_id;
#查看指定query_id的SQL语句各个阶段的耗时情况
show profile cpu for query query_id;
#查看指定query_id的SQL语句CPU的使用情况。

3、explain执行计划

EXPLAIN获取MySQL如何执行Select 语句的信息，包括select语句执行过程中如何连接和连接的顺序。

EXPLAIN SELECT 字段列表 FROM 表名 WHERE 条件;
#id 
#表示执行顺序，id相同从上到下，id不同，值越大越先执行。

#select_type
#select类型，常见类型：SIMPLE(简单表)	PRIMARY(主查询，即外层查询)	UNION(UNION的第二个或者后面的查询语句)	SUBQUERY(SELECT/WHERE之后包含了子查询)等

#type
#表示连接类型：性能由好到差连接类型为NULL,system，const，eq_ref，ref，range，index，all
#不查询任何表就是NULL
#根据主键或者唯一索引const
#使用非唯一索引就是ref
优化时保证尽量不要出现all

#possible_key
#这张表中可能使用到的索引，一个或者多个

#key
#实际用到的索引，Null则表示没有用到索引

#key_len
#表示索引中使用的字节数，该值为索引字段最大可能长度，并非实际使用长度，再不损失精度的前提下，越短越好。

#rows
#Mysql认为必须要执行查询的行数，在InnoDB引擎表中，是一个估计值，可能并不是准确的。

#filtered
# 表示返回结果的行数占需读取行的百分比，filtered的值越大越好。

#Extra
#额外信息字段

执行顺序：3、2、1、1

6、索引使用

最左前缀法则

联合索引，要遵循最左前缀法则。最左前缀法则指的是查询从索引的最左列开始，并且不跳过索引中的列。

如果跳过某列，索引将部分失效（后面的字段索引失效）。最左边的列必须存在，最左边字段不存在则索引全部失效。

走不走索引与字段位置无关

联合索引顺序性：

（123）col，联合索引定义

123，走索引

12，走索引

1，走索引

23，不走索引，全表扫描

3，不走索引

13，1走索引，3不走索引，遵循顺序原则

321，字段全部走索引，字段存在就行，没说必须放最左边，与位置无关。

范围查询

联合索引中，出现范围查询(<,>)，范围查询右侧的列索引失效。使用范围查询的字段不会失效

改进：使用>=或<=，不使用<，>。

123 col 联合索引定义
1，2(>/<)，3	12走索引，3不走
1，2(>=/=<),3 全部走索引

索引列运算

不要再索引列上进行运算操作，否则将会失效。

字符串不加单引号

字符串类型字段使用时不加单引号，索引将失效。

#1 type=char
where 1='1'   走索引
where 1=1     不走索引

模糊查询

如果只是尾部模糊查询，索引不会失效。如果是头部模糊匹配，索引失效。

where 1 like '软件%';		走索引
where 1 like '%软件';		不走索引
where 1 like '%软件%';	不走索引，只要前面加上模糊查询就不走索引。

or连接条件

or分割开的条件，or条件之前的字段有索引，后面的列没有索引，那么不会走索引。

where a=10 or b=20 
#a列创建了索引，b列没有索引，这种情况不会走索引。

where a=10 or b=20
#a列有索引，b列有索引，这种情况会走索引。

数据分布影响

如果Mysql走索引效率比全表扫描慢，那么Mysql会自动走全表扫描。

取决于你的数据分布情况和条件

如果数据中都是有值的，你查询条件是is null ，那么走索引

如果数据中都是有值的，你查询条件是is not null ，那么不走索引

如果数据中都是空值的，你查询条件是is null ，那么不走索引

如果数据中都是空值的，你查询条件是is not null ，那么走索引

SQL提示

优化数据库的一个手段，简单来说，在SQL中加入一些人为提示来达到优化的目的。

a字段是联合索引的最左列，同时还有一个单列索引。
在运行时，默认情况：优化器会索引中自动二选一。
use index：

告诉数据库你建议用哪个索引，也可能不会用这个索引（不接受你的建议）

explain
select * from table_name use index(index_name) where profession='软件工程'
ignore index：

不要用哪个索引。

explain
select * from table_name ignore index(index_name) where profession='软件工程';
force index：

你必须走哪个索引。强制索引

explain
select * from table_name force index(index_name) where profession='软件工程';

覆盖索引

尽量使用覆盖索引，（查询使用了索引，并且需要返回的列，在该索引中已经全部能够找到，避免回表），减少select *。

using where ,using index。	查询使用了索引，不需要回表查询，因为需要的值在二级索引中都可以拿到。
using index condition.		 查询使用了索引，因为二级索引中没有name字段值，所以需要回表查询。

前缀索引

字符串类型，varchar、text等，如果存储了很长的字符串，就会让索引变得很大，查询时，浪费大量的磁盘IO，影响查询效率。

可以只将字符串的一部分前缀，建立索引，大大节省存储空间，从而提高查询效率。

前缀长度

根据索引的选择性来决定，不重复的索引值（基数）和数据表中的记录总数的比值，索引选择性越高则查询效率越高，唯一索引的选择性是1，这是最好的索引选择性，性能也是最好的。

select count(distinct substring(col,1,5))/count(*) from table_name;
创建索引
create index index_name on table _name(column(n));
//列名后面加(n)，n代表提取这个字段的前面几个字符。
例子：(name(2))	name字段前2个字符建立索引。
查询过程

先截取字段一部分字符，去查询字段的二级索引，匹配后获取id值进行回表查询，获取数据，进行整个字段匹配，完全匹配就返回这行数据。不完全匹配就继续二级索引中查找下一个配置值，循环这个过程。

单列索引和联合索引

在业务场景中存在多个查询条件，针对查询字段建立索引，建立联合索引，而非单列索引。主要还是避免回表查询。单列索引很容易就回表查询了。

7、索引设计原则

  1. 针对数据量较大，查询频繁的表建立索引。超过100万再说。

  2. 针对常作为查询条件，排序，分组操作的字段建立索引。尽量使用联合索引。

  3. 选择区分度高的列作为索引，建立唯一索引，区分度越高，使用的索引效率越高。

  4. 字符串类型的字段，字段长度较长，针对他们的特点，可以建立前缀索引

  5. 尽量使用联合索引，减少单列索引，查询时可以覆盖索引，避免回表操作，节省空间，提高查询效率。

  6. 控制索引数量，索引越多，维护代价就越大，会影响增删改查的效率。

  7. 如果索引列不能存储Null值，请在创建表时使用 NOT NULL 来约束他。优化器知道每列是否包含Null值时，他可以更好的确定哪个索引最有效的用于查询。

面试题

存储引擎有哪些，有什么区别？

innoDB支持事务，MyISAM不支持

innoDB支持行锁，MyISAM支持行锁

innoDB支持外键，MyISAM不支持外键

为什么innoDB选择B+Tree索引结构？为什么不选择其他结构？

相比二叉树，层级更少，搜索效率更高

相比B树， 一页中存储的键值减少，指针跟着减少，要同样保存大量数据，只能更加树的高度，导致性能降低。

相比红黑树，

相比原B+树，增加了叶子节点指针，形成了双向链表，便于范围搜索和排序。

为什么要用B+树索引而不是用Hash索引？

对于hash索引来说不支持等值匹配，不支持范围匹配和排序操作的。

InnoDB主键索引B+Tree高度有多高？

假设1行数据是1k，一页可以存储16行这样的数据。InnoDB指针占用6个字节的空间，主键为bigint，占用字节数为8。
